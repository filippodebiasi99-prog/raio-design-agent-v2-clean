# 11 · Mini Sample Protocol

Use this protocol while calibrating RaiO visual quality.

## Why

Full dashboards make Codex focus on too many things:
- data
- layout
- responsive
- rail
- copy
- charts
- tables
- paths
- content completeness

This hides the real question:
Is the visual foundation good?

## Mini sample scope

For visual tests, create only:

1. One compact focus module
2. One 3-row lead list
3. One small right/context card

No full dashboard.
No large sidebar.
No complete app shell.
No big table.
No complete responsive product.
No metric grid.
No extra widgets.

## Goal

The goal is to test:
- typography
- palette
- spacing
- card treatment
- button treatment
- row treatment
- one custom visual detail

## Required before coding

Before coding the mini sample, output:

```md
## Mini Sample Contract

Selected references:
Primary visual move:
Typography foundation:
Palette foundation:
Component foundation:
Custom enrichment:
Asset need:
What I will NOT build:
Why this will not become a generic Codex dashboard:
```

## File output

Create output outside the agent repo.

Do not create tests inside the agent repo unless explicitly requested.

## Failure

If the result looks like:
- a generic small dashboard
- a normal card/list/card layout
- Codex with a new font
- a prompt-generated SaaS sample

then it failed.

Mini does not mean boring.
Mini means focused.
