# 01 · Reference-First Workflow

Before designing, choose a reference stack.

## Required reference stack

Choose 2-3 references and assign roles:

### Foundation reference
Controls:
- overall cleanliness
- credibility
- density
- baseline component quality

### Component reference
Controls:
- table/list/card/button/input quality
- spacing
- rows
- pills
- navigation

### Custom/detail reference
Controls:
- optional custom visual language
- SVG
- map
- chart
- timeline
- preview
- pattern
- image richness

## Example for RaiOS

```md
Foundation reference: Reference 10 · Clean Admin Baseline
Reason: production-like cleanliness, spacing, controls and admin credibility.

Component reference: Reference 05 · Premium CRM List
Reason: compact rows, small typography, status pills, avatar/entity details.

Custom/detail reference: Reference 07 · Soft Purple Learning Dashboard
Reason: home-module richness, controlled decorative detail, stronger arrival feeling.
```

## Do not choose objects

Bad:
- I choose reference 03 because it has a map.
- I choose reference 07 because it is purple.
- I choose reference 02 because it has black sidebar.

Good:
- I choose reference 03 because product mechanics are visualized as custom micro-UX.
- I choose reference 07 because typography, color field and card rhythm are calibrated.
- I choose reference 02 because navigation/illustration are treated as proprietary visual language.

## Reference fidelity

The output must visually feel related to the selected reference stack.

If the result does not resemble the selected reference quality, it fails.

## Required extraction

For each selected reference, extract:
- typography principle
- palette principle
- spacing/component principle
- custom/detail principle
- what not to copy literally
