# 00 · Operating Principle

## Why V2 exists

V1 produced many useful rules, but Codex treated them as a checklist.

It would create:
- lead queue
- selected lead rail
- score
- path
- green accent
- asset brief

and then call the result 8/10.

That is not enough.

## The new principle

Do not start from components.
Do not start from custom objects.
Do not start from dashboard defaults.

Start from the references and the foundation:
1. Typography
2. Palette
3. Spacing and component system
4. Custom enrichment

The user does not primarily like maps, sidebars, SVGs, cards, logos or timelines.

The user likes when these things are:
- beautifully typeset
- well spaced
- color-calibrated
- product-specific
- visually intentional
- not default
- not template-like

## Core command

Do not copy the object.
Extract the execution quality.

## Rule compliance is not visual quality

Having the right elements does not mean the UI is good.

A UI can contain:
- score
- path
- lead table
- preview
- SVG
- asset brief

and still be ugly.

The visual quality is judged by:
- typography
- palette
- spacing
- component finishing
- reference fidelity
- first impression

## No default Codex aesthetic

Codex default dashboard aesthetics are not allowed unless they match the selected reference stack.

If the result looks like "clean AI dashboard" rather than something grounded in the reference set, it fails.
